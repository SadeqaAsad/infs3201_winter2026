A4 Refactor Plan


1. Collections That Will Change

- employees: Remove the "employeeId" field (e.g. "E001"). The MongoDB _id will be used as the primary key instead.
- shifts: Remove the "shiftId" field (e.g. "S001"). Add an "employees" array that embeds the ObjectIds of assigned employees directly inside each shift document. The MongoDB _id will be used as the primary key instead.
- assignments: This collection will be removed entirely. Its data will be migrated into the shifts.employees embedded array.
- users (new): A new collection will be created manually in Compass containing username and sha256-hashed password for authentication.
- security_log (new): A new collection will store a log entry for every HTTP request made to the application.

2. Persistence Functions That Must Be Modified or Removed

- getAllEmployees(): No change needed.
- findEmployee(employeeId): Must change to findEmployee(id) where id is a MongoDB ObjectId string. Must use new ObjectId(id) for the query instead of { employeeId: id }.
- saveEmployees(): Must be updated to no longer set an employeeId field.
- updateEmployee(employeeId, ...): Must change to use _id ObjectId instead of employeeId string.
- getAllShifts(): No change needed.
- findShift(shiftId): Must change to use _id ObjectId instead of shiftId.
- getAllAssignments(): Remove this function. Assignments are now embedded in shifts.
- getAssignmentsByEmployee(employeeId): Remove this function. Employee schedule is now obtained by querying shifts where the employees array contains the employee ObjectId.
- getAssignmentsByEmployeeAndDate(): Remove this function. No longer needed.
- New function needed: getShiftsByEmployee(employeeObjectId) - queries shifts collection where employees array contains the given ObjectId.

3. Required Changes to Business Logic Functions

- getEmployeeSchedule(employeeId): Must be refactored to call the new getShiftsByEmployee() persistence function using ObjectId instead of the old assignment-based lookup.
- findEmployee(employeeId): Must pass an ObjectId to persistence layer instead of a string employeeId.
- updateEmployee(): Must pass ObjectId to persistence layer.
- All places in app.js that use req.params.id must pass it to business functions that convert it to ObjectId internally.

4. How Embedded Arrays Affect Update Operations

- Previously, adding or removing an employee from a shift meant inserting or deleting a document in the assignments collection.
- Now, adding an employee to a shift means using $push on the employees array inside the shift document.
- Removing an employee from a shift means using $pull on the employees array.
- Querying which shifts an employee is assigned to now requires a query like: db.shifts.find({ employees: ObjectId("...") }) instead of a join-style lookup through assignments.
- This is more efficient for reads but means shift documents grow slightly with each employee added.

5. Risks and Complications

- ObjectId vs string: All code that previously compared employeeId or shiftId as strings must be updated to use ObjectId comparisons. Passing a plain string where an ObjectId is expected will silently return no results.
- Migration order matters: The transform_db.js steps must run in order. Embedding employees into shifts (Step 2) must happen before removing the assignments collection (Step 3), otherwise the data will be lost.
- No rollback: Once the assignments collection is dropped and employeeId/shiftId fields are removed, the migration cannot be easily reversed without restoring from a backup. A database backup should be taken before running transform_db.js.
- Session management without express-session: Sessions must be implemented manually using cookies and a sessions store (e.g. an in-memory object or MongoDB collection). The 5-minute timeout must be enforced and extended on every request.
- Cookie security: Without express-session, care must be taken to generate secure, unpredictable session tokens.
